## 5. Enseignements méthode

### 5.1 Repo-informed prompt generation

ChatGPT devrait **consulter Git** (status, diff, log, fichiers cibles) **avant** de produire le prompt Cursor, plutôt que de s'appuyer uniquement sur le résumé conversationnel.

**Recommandation candidate (PR 2) :** section obligatoire « Local Git Truth Check » + « fichiers à lire » dans le flux ChatGPT amont.

### 5.2 Git comme review backend

Quand ChatGPT a accès au repo, `git diff --stat`, `git diff --name-status` et `git log` peuvent **remplacer partiellement** les embeddings massifs du review pack.

**Recommandation candidate (PR 2) :** introduire un **Git Review Index** — liste structurée des fichiers et sections pertinentes, sans duplication intégrale.

### 5.3 Review pack proportionné

Le review pack ne devrait pas être systématiquement **full**. Niveaux candidats :

| Niveau | Quand | Contenu minimal |
|--------|-------|-----------------|
| **none** | Micro-fix technique, typo, E2E locator | Rapport Cursor court uniquement |
| **light** | Delivery borné, 1–5 fichiers | Git truth + diff stat + résumé |
| **full** | Cycle documentaire structurant, capitalisation, clôture | Sources, extraits, document complet, gates |

**Recommandation candidate (PR 2) :** aligner sur profils Light / Standard / Capitalization du operating model.

### 5.4 Figma visual contract

Figma-to-code exige un **contrat visuel explicite** avant delivery :

- références Figma (fileKey, node-id, frames)
- grille d'écarts structurels **et** visuels
- **capture runtime obligatoire** avant tout verdict `NO DELIVERY REQUIRED`
- comparaison MCP Figma + screenshot Morris

**Recommandation candidate (PR 3) :** standard « Figma visual contract » + règle qualité gap review (déjà capitalisée dans gap review R-UX-01 §7 bis).

### 5.5 Gates Morris fréquents, Git regroupé

Les **gates Morris** restent fréquents sur les décisions structurantes ; les **opérations Git** (commit, push, PR) devraient être **regroupées** quand le périmètre est stable.

**Recommandation candidate (PR 4) :** définir un commit unique par cycle UX/documentaire quand possible ; limiter les micro-PR post-merge sauf nécessité.

### 5.6 Stop rules post-MVP

Une phase post-MVP peut **s'arrêter** lorsque :

- les réserves structurantes ciblées sont traitées ou explicitement reportées ;
- les réserves mineures/environnementales restent **ouvertes mais acceptées** ;
- la matière REX est suffisante pour capitalisation ;
- Option B/C ne sont pas lancées sans GO Morris distinct.

**Recommandation candidate (PR 4) :** formaliser les **conditions d'arrêt post-MVP avec réserves acceptées**.
